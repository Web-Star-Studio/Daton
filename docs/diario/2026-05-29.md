### 3. Construção e validação do pipeline ETL v1 → v2
- **O que:** desenhado um pipeline completo de migração da base LAIA do projeto v1 (Supabase Postgres) para a v2 (Neon Postgres / Drizzle), composto por seis arquivos em `scripts/migration-v1-to-v2/`:
  - `01-staging-schema.sql` — DDL idempotente que cria o schema isolado `_migration` com tabelas espelho do v1 e auxiliares (`id_map` para pivot UUID → serial, `skipped` para auditoria de órfãos, `config` para parâmetros).
  - `02-transform.sql` — ETL transacional que resolve usuários por e-mail, unidades por código, setores via `id_map`; mapeia enums (`ativo` → `active`, `significativo` → `significant`, `finalizada` → `finalized`); deriva booleans (`control_level`, `normal_condition`); converte soft-delete v1 (`deleted_at`) em arquivamento v2 (`archived_at` + `purged_at` nulo); e expande o JSONB `legislation_references` em linhas normalizadas em `laia_requirement_links`.
  - `03-validate.sql` — checagens de contagem v1 vs v2, integridade de FK e distribuições.
  - `04-rollback.sql` — desfaz exatamente o que foi migrado, sem tocar em dados nativos.
  - `dry-run.mjs` — driver com `BEGIN ... ROLLBACK` transacional que injeta mock sintético e roda 23 asserções sobre forma dos dados migrados.
  - `README.md` — guia operacional.
- **Validação local:** pipeline executado contra um Postgres local em Docker (com schema v2 aplicado), com **23/23 asserções verdes** cobrindo mapeamento de usuário/unidade, contagens, enums, booleans derivados, expansão JSONB → normalizada, fallback de usuário, soft-delete → lixeira, revisão finalizada e idempotência (segunda execução não duplica). Confirmado o `ROLLBACK` completo (contagens voltam a zero).
- **Por quê:** validar exaustivamente a transformação contra um ambiente espelho **sem nenhum toque em produção**, antes de migrar dados reais.
- **Impacto/área:** novo diretório `scripts/migration-v1-to-v2/`.
- **Status:** concluído.

  - Ajustes ao transform durante o processo:

- **Resultado:**
